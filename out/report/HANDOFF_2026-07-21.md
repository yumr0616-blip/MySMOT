## 3. 逐阶段详述

### 3.1 迁移收尾（Phase A，`0884f47` + `f6410e9`）

- **环境重建**：旧 `.venv` 是 Windows 下 `C:\Python312` 创建的产物，在 Linux 上不可用，删除重建。系统无 conda/uv/pip3，先装 uv，再 `uv venv --python 3.12`，按旧环境精确版本钉住依赖：`torch==2.11.0+cu128 transformers==5.12.1 accelerate==1.14.0 bitsandbytes==0.49.2` 等（当前实测版本见 §7）。cu128 轮子同时覆盖 Blackwell(sm_120) 与 Ada Lovelace(sm_89)，所以 README 里那条安装命令不用因为换了卡型号而改。
- **仓库卫生**：新增 `.gitattributes`（`* text=auto`），`git add --renormalize .` 后把工作区强制对齐 index，消灭了 30 个文件的 CRLF/LF 噪音（`git diff --ignore-cr-at-eol` 在此之前就已确认这些"改动"是纯噪音，不是真实 WIP）。
- **冒烟验证三连**：① pytest 全套；② `python -m smot.ml.gradient_check --quantize-4bit`（当时还是旧 2B 底座）GATE PASS，顺带验证代理下载/CUDA/bitsandbytes 三件事都通；③ 用现存 `out/stage1b/stage1b.pt` 在新机器上跑 8 段评测，和旧机器 `out/eval8_*` 的数字逐位比对，确认迁移无损（注：跨 GPU 架构的浮点非确定性导致生成文本本身会有出入，但确定性的前端流水线统计完全一致）。

### 3.2 底座切换：27B 冲了一次，退回 9B（`908ef7d`）

**计划**：`DEFAULT_MODEL_ID` 换成 `Qwen/Qwen3.5-27B` + bnb nf4 量化（bf16 权重约 56GB 装不下，nf4 后约 15-18GB）。

**实际发生的事**：qwen3.5 是混合线性注意力/全注意力架构（HF 里的 `qwen3_5`），它的 `linear_attention` 层要吃满性能需要 `flash-linear-attention` + `causal-conv1d` 两个包提供的融合核；`causal-conv1d` 需要系统级编译 CUDA 扩展，而这台机器没有 sudo、没有系统 CUDA 工具链，装不上。缺了融合核，这些层的反向传播退回朴素 PyTorch 实现（`torch_chunk_gated_delta_rule`），这条回退路径的反传峰值显存远超权重体积本身暗示的量级。

排查/尝试过的方案，均未解决：
- `--grad-checkpoint`（梯度检查点）：对这个瓶颈基本无效——瓶颈不是"跨层可复用的激活值"，而是单层反传内部的峰值。
- `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`：缓解的是分配器碎片化，不是这类问题。
- 把预算收到最紧（640px、top-k 2）：仍然 OOM。
- 只装 `flash-linear-attention`（`causal-conv1d` 依旧缺失）：部分生效，但快速路径从未完全启用。

**决策点**：向用户汇报后，用户选择"回退 Qwen3.5-9B bf16（推荐）"而非继续攻 27B。9B 用 bf16 即可（约 18GB，单卡装得下不需要量化），`--quantize-4bit` 开关保留作为进一步压缩显存的可选项（例如以后换更大底座，或多进程共卡时）。

**同一提交里做的预算放开**：`--top-k-frames`/`--top-k-pair-frames` 2→4（与推理侧 `PipelineConfig` 对齐，训推同预算）、`--image-max-side` 640→1024（后因 9B 上仍 OOM 进一步降到 800，见下）、帧渲染缓存 `cache_size` 2→8、`empty_cache_every` 20→100。

**9B 在 1024px 上仍 OOM**，用 B-gate 吞吐/显存探针实测确认后降到 **800px**——这是当前生效的默认值，配合 `--grad-checkpoint` 使用（900px 附近没有细测，800 是验证过安全的档位，不代表精确上限）。

### 3.3 关键 bug：thinking mode 吃光整个生成预算（`b7f49ba`）

这是本轮**唯一一个会让下游数字彻底失真**的 bug，值得单独强调。

**现象**：换到 9B 后，用同样的预算跑基线评测，interaction F1 **恒为 0**——不是变差，是所有口径（连最宽松的 coarse）都零匹配。

**根因排查过程**：没有停留在"模型变差了"的猜测上，而是直接打印了模型的真实生成文本——发现输出是大段推理散文，从未出现过要求的结构化 JSON。再去看 `chat_template.jinja` 的源码，确认 qwen3_5 的模板默认会在生成头（`<|im_start|>assistant\n`）之后自动留一个未闭合的 `<think>\n`，指望模型自己写一段思维链再 `</think>` 收尾才正式作答。而这里给的生成预算只有 96 个 token，根本不够写完这段思考——模型的输出从头到尾都卡在"思考中"，结构化答案从未出现，OutputAssembler 自然一条都解析不出来。

**修复**：在全部三处 `apply_chat_template(..., add_generation_prompt=True)` 调用（`smot/ml/qwen_adapter.py` 里的 `teacher_forced_loss()` 一处 + `QwenMLLMAdapter.generate()` 一处 + `soft_token_position()` 内部一处不需要改因为它不带 generation prompt）显式传入 `enable_thinking=False`。修复后模板直接吐 `<think>\n\n</think>\n\n`，模型从干净的作答语境起步。

**验证**：先用 3 条序列做冒烟测试确认 JSON 正常出现，再重新跑了完整的 100 段基线评测。修复后的数字：未训练 0.160 → 1b·1ep 0.273 → 1b·2ep 0.372（提升幅度与 2B 阶梯相近；未训练档 strict 略低于 2B 的 0.199，但 synonym_merged 档反超，判定为同义词表锁定在 2B 词汇分布导致的比较假象，不是真实语义变差——两个底座的 base 数字本就不适合直接比较）。

**这个 bug 出现时机的教训**：如果不是主动去读模型的原始生成文本、而是只看聚合后的 F1=0 就下结论，很容易误判成"9B 比 2B 差"或者"评测代码坏了"。以后任何底座/框架升级后出现指标断崖式下跌，第一步应该是看几条原始生成样例，而不是先怀疑评测逻辑。

### 3.4 P2-1..4：视觉塔特征进 KFA（`3d9d9ab` + `795fc57`）

**目标**：两个 KFA（Key Frame Attention，把轨迹几何量转成给 MLLM 的 soft token 的可学习打分模块）的打分输入从纯几何升级为几何+外观。此前纯几何偶尔会选中被遮挡的帧（示例序列里有实证）。

**方案选择**：设计文档 §4 预留了两个候选方案——A（复用冻结 Qwen 自带的视觉塔，无新依赖，但需要按帧抽样控制成本）和 B（轻量冻结 CLIP，快但引入新依赖、特征分布与底座脱钩）。**方案 A 一次命中**，没有走到方案 B。

**新增 `smot/ml/feature_cache.py`（319 行）**——离线逐序列建 `(track_id, 帧号) -> 1152 维向量` 的缓存：
- 特征来源：`model.model.visual`（Qwen 视觉塔），取 **merger 之前**的 `last_hidden_state`（`hidden_size=1152`，比 merger 之后的 `out_hidden_size=4096` 省一半以上），对每个 track 在某一帧的 bbox crop 做前向，patch token 做 mean pool。
- 采样策略：unary（每帧都要打分）按 `stride=6` 稀疏采样 + 最近邻回填（`NearestFillCache`）；pairwise 只需要 `EventCandidateFilter` 选出的候选帧，本就稀疏，理论上精确命中（仍留最近邻兜底防止训练/推理两侧候选帧计算因参数漂移不完全一致）。两者共享底层缓存，同一个 crop 不会算两次。
- 存盘：每条序列一个 `.npz`，键 `"{track_id}_{t}"`。

**接线**（`795fc57`，改 `examples/run_bensmot_real.py` / `smot/ml/gradient_check.py` / `smot/ml/training.py` 三处）：新增 `--feat-cache-dir` CLI 参数（默认 `None`，不给就是纯几何的 Stage-1a/1b 行为，完全向后兼容）；给了这个参数后，`frame_feature_fn`/`pair_feature_fn` 改用 `augmented_frame_features`/`augmented_pair_feature_vectors`（几何向量后面拼视觉向量），两个 KFA 类本身**不需要改代码**——`in_dim` 从一开始就是普通构造参数，9→1161（`FRAME_FEATURE_DIM + 1152`）、8→2312（`PAIR_FEATURE_DIM + 2*1152`）纯粹是构造时传的数字不同。梯度门禁加了 `--stage2` 开关复用同一套四模块检查逻辑。

**验收结果**：120 段·1 epoch·seed 0，100 段评测口径 strict F1 **0.409**，远超验收线（Stage-1b·1ep 的 0.273），甚至超过纯几何 Stage-1b·2ep 的 0.372（同一 9B 底座）。证据帧质量代理指标（选中帧 visibility 均值）未能验证——这 100 段测试集选中帧的 visibility（gt 第 9 列）全部恒为 1.0，数据本身没有遮挡信号可对比，不是方法失效，是这批测试数据凑巧没覆盖到这个场景。

### 3.5 P3：全量训练规模扩展（无独立提交，纯 `out/` 产物 + 文档）

用户在看到 Stage-2 结果后明确要求："现在有一整个晚上的时间，我们跑一下全量吧"——把训练规模从 120 段扩到全量 2186 段（BenSMOT train 集 2279 段中 93 段因标注缺失被 `--skip-errors` 过滤掉）。

**特征缓存全量构建**：879MB（train 2186 段 + test 100 段全覆盖）。

**训练**：Stage-2 全量 2 epoch，约 18500 步。100 段评测口径 strict F1：
```
120段·1ep 0.409 → 全量·1ep 0.434（本轮最优） → 全量·2ep 0.421（小幅回落）
```
2ep 相对 1ep 精确率下降、预测数增多，判断是过拟合边缘或噪声，没有做多种子区分去坐实。**规模提升方向确认成立**（更多数据确实带来提升），但幅度不大（+0.025，1ep 档），说明"下一个数量级的提升"更可能来自架构改进而不是单纯堆数据规模。

**基础设施问题（本节是本轮最大的一处"教训"，值得完整记录）**：全量训练最初按"单次连续后台跑"的方式启动，结果被外部反复终止，一共 **4 次**，间隔从几分钟到 88 分钟不等。逐一排查了：`dmesg`、`journalctl`、`syslog`、`kern.log`、`systemd-oomd`、GNOME 自动挂起设置、`crontab`、`ulimit`、看门狗进程——**没有找到任何根因**，怀疑是这个会话/沙盒环境层面的限制，属于已知但未解决、且可能无法从这一层观测到原因的问题。

期间三次就"要不要继续跑、怎么跑"征询用户，用户依次给出明确方向：确认不是用户自己手动停止的 → 白天直接重跑 → 发现白天也照样被杀之后，改成"小步长分段跑（每次 500 步左右）"。**最终生效的绕过方案**：不用 `run_in_background`，改成前台执行 + `--max-steps <目标步数> --resume`（约 500 步一段，配合 `--save-every 50`，每段设 600 秒超时），连续跑了 20+ 段全部正常完成。分段跑期间还遇到过 2 次真实的 CUDA OOM（不是被杀，是真报错，完整 traceback 指向变长真实语料样本导致的显存碎片化），加上 `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` 后未再复现，配合简单重试即可（重试时分配器状态是新鲜的）。

`next_steps.md` 里已经把这条经验记成"下一轮长跑（若涉及）应默认采用分段模式，不要假设单次连续后台跑能扛住数小时"——这是留给以后所有长跑任务的操作性结论，根因本身仍然未知。

### 3.6 多种子复核（无独立提交，`out/eval100_9b_stage2_s1` / `s2` + `next_steps.md` 更新）

全量结果出来后，用户被问及"接下来还有什么可以做的"，选择"先验证现有结论"——即先把 Stage-2 的核心断言坐实，而不是急着扩展新范围。复用同一份 120 段·1ep 的 feat_cache，只换 `--seed 1`/`--seed 2` 重训两次：

```
seed0 0.409（原始）  seed1 0.325  seed2 0.408
均值 0.381，极差 0.084
```

**结论**：核心断言站得住——即便取三个种子里最差的 0.325，仍明显高于 Stage-1b·1ep 单种子的 0.273（+0.052）；但 0.409 这个具体数字不该被当成精确值，更诚实的表述是"约 0.38 ± 0.04"。**这个比较仍不完全对称**：Stage-1b 一侧的 9B 种子波动从未测量过（只对 Stage-2 做了多种子），这是一处已知但未处理的不对称，不在这次验证的范围内，也不是用户已要求解决的事项。

### 3.7 可视化产出（`out/report/make_report.py`，不进仓库，但这是本轮明确要求的交付物）

### 3.8 P4 前置：IoU 轨迹匹配（`0815d4f`）

全量结果验证完之后，用户说"下一步吧"，选择推进此前已提出的另一个选项：P4（真实跟踪器接入）的前置工作。

**问题背景**：`smot/eval.py` 的交互 F1 / instance 覆盖率全靠 `subject_id`/`object_id`/`track_id` **直接相等**配对预测和 gold。当前 `StubTracker` 直接读 GT 轨迹，预测 id 天然等于 gold id，这个假设成立；一旦换成真实检测+跟踪器（YOLO/RT-DETR+ByteTrack 或 SAM2），跟踪器会分配自己的一套 id，与 gold id 毫无关系，直接相等匹配会整体失效——所有指标会看起来"跟踪器很差"，即使语义理解本身没问题，这其实是在测两件不同的事。

**新增 `smot/track_matching.py`（149 行，纯 stdlib，不放进 `smot/ml/`）**：
- `_track_iou()`：轨迹级 IoU = 逐帧 box IoU 在两条轨迹观测帧号**并集**上取均值（不是交集——用交集会让"仅一帧巧合重叠"被误判成满分匹配；并集分母下，共同观测帧占比低的短暂重叠自然被拉低分数）。
- `match_tracks()`：贪心匹配（用户选择，非 Hungarian，不引入 scipy 依赖，单视频轨迹数通常是个位数，贪心和最优解在这个规模下几乎不会给出不同结果）。轨迹级 IoU 达不到阈值（默认 0.5）的最高分也不接受，宁可算未匹配也不牵强凑对。
- 未匹配的预测轨迹计 FP、未匹配的 gold 轨迹计 FN（用户选择：新增一个独立的 tracking 指标块，不混进现有语义层 F1），`TrackMatchResult.stats()` 输出 track 级 precision/recall/F1 + 匹配上那部分的平均 IoU。
- `remap_assertion_ids()`：集成点——真实 tracker 接入后，先用它把预测断言里的 id 重写成匹配到的 gold id，再走现有 `smot.eval` 全套逻辑，**`eval.py` 本身不需要改一行**。

**验证**：12 个合成单测（恒等匹配、跨 id 匹配、FP 幻觉轨迹、FN 漏检、低于阈值拒绝匹配、贪心优先级、`stats()` 数值、`remap_assertion_ids` 的三种边界情况）全过；另在 5 条真实 BenSMOT 序列上做了 `match_tracks(seq.trajectories, seq.trajectories)` 的合理性检查——当前 StubTracker 场景下匹配结果是完美恒等映射，`mean_matched_iou=1.0`，确认相对现状零回归。

**明确仍未做的事**：tracker 本身的选型/接入（YOLO/RT-DETR+ByteTrack 还是 SAM2）依旧是"更远期"项，这次只完成了它明确点名的前置工作，没有涉及实际接入。

## 6. 环境与基础设施踩坑

按遇到的先后顺序，每条都是"以后再犯就是没吸取教训"级别的坑：

1. **git 提交人身份缺失**：`git commit` 报"作者身份未知"。用 `git log` 看仓库既有作者约定，`git config --local`（不是 `--global`）对齐成 `yujinwen <120868466+YuuJW@users.noreply.github.com>`。
2. **httpx/huggingface_hub 在 SOCKS 代理下报 `Unknown scheme for proxy URL socks://...`**：环境里 `ALL_PROXY`/`all_proxy` 是 `socks://` 而不是 `socks5://` scheme，httpx 不认。凡是要碰 HF 网络的命令一律前缀 `env -u ALL_PROXY -u all_proxy`（`HTTP_PROXY`/`HTTPS_PROXY` 是合法 `http://` scheme，留着不受影响）。
3. **`hf_xet` 下载极慢**（实测约 100KB/s，vs 裸 HTTP 约 4.3MB/s）：用 `curl --range` 做速度对比确认后，设 `HF_HUB_DISABLE_XET=1` 强制走普通 HTTP 下载器。
4. **27B + nf4 在反传时 OOM，且 `--grad-checkpoint`/`expandable_segments` 都救不了**：根因是 qwen3_5 的 `linear_attention` 层缺 `flash-linear-attention`+`causal-conv1d` 融合核（后者需要系统级编译，这台机器没有 sudo/CUDA 工具链），回退到朴素 PyTorch 实现后反传峰值显存远超权重体积暗示的量级。这不是"调参能解决"的问题，除非能装上 `causal-conv1d`，否则 27B 在这台机器上不可行。详见 §3.2。
5. **9B 在 1024px 图像分辨率下仍 OOM**：不是 bug，是真实显存天花板，用 B-gate 探针实测确认后降到 800px。
6. **qwen3_5 chat template 默认开启"思考模式"，96 token 生成预算全被推理散文吃光，F1 恒为 0**：本轮唯一严重 bug，详见 §3.3。**教训**：指标断崖式下跌时，第一步应该去看模型的原始生成文本，而不是先怀疑评测代码。
7. **`make_report.py` 里一处过长副标题字符串导致画布尺寸计算溢出**：通过实际跑脚本、肉眼检查渲染出的 PNG 发现并修掉的，纯靠读代码不容易看出来——生成可视化产物类的脚本，写完之后务必实际跑一遍看输出,不能只看逻辑"应该没问题"。
8. **后台长跑training进程被间歇性外部终止，根因未查明**：4 次终止，`dmesg`/`journalctl`/`syslog`/`kern.log`/`systemd-oomd`/GNOME 挂起设置/`crontab`/`ulimit`/看门狗进程全部排查无果，怀疑是会话/沙盒环境层限制。**实际有效的绕过方案**（不是根治）：放弃 `run_in_background`，改用前台 + `--max-steps ~500 --resume --save-every 50`（约 600 秒超时/段）分段跑,验证有效跑完 20+ 段。以后任何预计要跑数小时的任务默认按这个模式规划,不要假设单次连续后台跑能扛住。
9. **分段跑期间遇到的 2 次真实（非被杀）CUDA OOM**：变长真实语料样本导致显存碎片化，加 `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` 后未再复现,配合简单重试即可(重试时分配器状态是新鲜的)。

## 8. 已知问题与未解决项

- **后台进程被间歇性终止的根因始终未查明**（§3.5、§6第8条）——只有可靠的操作性绕过方案,没有根治。如果以后同样的现象在其他类型任务（不只是训练）上出现,值得留意是否是同一机制。
- **Stage-1b 一侧的 9B 种子波动从未测量**（§3.6）——Stage-2 做了 3 种子,Stage-1b 没有,两者的比较眼下是"多种子 vs 单种子"，不完全对称。如果要下更强的结论（比如"Stage-2 相对 Stage-1b 的提升在统计上显著"）,需要先补 Stage-1b 的种子。
- **证据帧质量的代理指标（visibility）在当前 100 段测试集上没有区分度**（§3.4）——不是方法失效,是这批测试数据凑巧没有遮挡样本可用于对比,但也意味着"Stage-2 减少了选中遮挡帧"这个原始设计动机,目前没有定量证据支持,只有 §3.7 三个示例卡的定性佐证。
- **全量 2ep 相对 1ep 的小幅回落（0.434→0.421）没有做多种子区分**（§3.5）——不确定是真实的过拟合信号还是种子噪声。
- **P4 的实际 tracker 选型/接入完全没有开始**（§3.8）——只做了前置的 id 匹配模块,YOLO/RT-DETR+ByteTrack vs SAM2 的选型、真实检测噪声对语义层分数的影响幅度,都还是空白。
